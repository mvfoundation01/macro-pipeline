## 2. Root cause analysis

**Primary cause**: Codex's fresh-clone environment lacks the data caches that Track A's persistent worktree accumulated over Layer 1+ runs. Specifically:

1. **Shiller series caches** (CAPE, CPI, dividend, earnings, GS10): generated by `macro_pipeline.loaders.shiller` on first invocation.
2. **FRED series caches** (many indicator pulls): generated by `macro_pipeline.loaders.fred_loader`.
3. **Yahoo / TV CSV / NAAIM / FINRA / NY Fed / Damodaran / AAII / Atlanta Wage / CFTC / HLW**: each loader writes parquet + meta-json cache pairs.

**Secondary cause**: pytest collection-time imports trigger loader execution for some tests (those with module-level fixtures or `pytest.fixture(scope="module")` calls that materialize caches at collection). Without caches, the import fails and pytest abandons the collection.

**Why `-x` stops the run**: `-x` (exit on first failure) propagates collection errors as test failures. Codex's report of stopping at `test_d27` is consistent with `test_ap6_narrowing.py::test_d27_site_still_propagates_regime_classifier_error_post_consolidation` (alphabetical/module-load order) triggering a cache-dependent collection import error.

## 5. L6-G measurement-registry deferred split — empirical verification

Codex Finding #6 also cited that Track A's L6-G claim of **"32 L7 / 18 L8a"** deferred entries didn't match Codex's count of **"36 L7 / 14 L8a"**. Empirical verification at L6-J (via `validate_registry_counts()`):

```
$ python -c "from macro_pipeline.ensemble import load_metrics_registry, validate_registry_counts; \\
             counts = validate_registry_counts(load_metrics_registry(), \\
                                                expected_computed=None, expected_deferred=None); \\
             print(counts)"
{'computed': 40, 'deferred': 50, 'deferred_to_l7': 36, 'deferred_to_l8a': 14, 'total': 90}
```

**Codex's count was correct**: actual split is 40 computed + 36 L7-deferred + 14 L8a-deferred = 90. Track A's L6-G claim of "32 L7 / 18 L8a" was a counting error in the L6-G commit message (the 40-computed + 50-deferred totals were correct, but the L7/L8a sub-split was misstated).

The L6-J `validate_registry_counts` helper now enforces these counts at validation time:

```
validate_registry_counts(registry, expected_computed=40, expected_deferred=50)
```

Future drift will raise `ValueError` at validation, preventing silent regression of the registry composition.
